# RFC 7517 — JSON Web Key (JWK)

The simplest practical key-history publication format — useful as the "verification side" companion to whatever envelope encoding (CBOR/JSON/protobuf) the device uses on the signing side.

## JWKS format

```json
{"keys": [JWK, JWK, ...]}
```

JSON, no inherent ordering.

## `kid` purpose (direct quote)

`kid` is explicitly designed for "choose among a set of keys within a JWK Set during key rollover."

## Rotation pattern

1. Publish current + previous keys simultaneously
2. Signers stamp the active `kid`
3. Verifiers select by `kid` from the JWKS

## Ed25519 in JWK

`kty: "OKP"` with `crv: "Ed25519"` per RFC 8037.

## Limitations

- JWKS has **no versioning of its own** — it's a live document; clients re-fetch to discover new keys
- Edge devices typically pin a JWKS URL or cache it with TTL
- **No threshold or chain-of-trust** — assumes a single trusted publisher
- Rotation safety depends entirely on transport (TLS) and freshness
