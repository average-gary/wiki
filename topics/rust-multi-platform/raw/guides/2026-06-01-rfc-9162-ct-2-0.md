# RFC 9162 — Certificate Transparency v2.0 (Experimental)

Obsoletes RFC 6962 (2013). Defines the canonical Merkle tree primitives used by every modern transparency log.

## Hash structure

- Leaf hash: `HASH(0x00 || data)` — domain-separation tag prevents second-preimage attacks
- Internal hash: `HASH(0x01 || left || right)`
- Tree shape "uniquely determined by the number of leaves" even when not a power of two; recursive split at largest power-of-two `k < n`

## Signed Tree Head (STH)

- Carries: log_id, ms-precision timestamp, tree size, root hash, extensions, signature
- "Each log SHOULD sign its current tree head information"
- STH is itself a signed envelope — one of the smallest, most-deployed signed-envelope formats in existence

## Proofs

- **Inclusion proof** — shortest list of nodes sufficient to recompute the root containing a given leaf
- **Consistency proof** — bound at `ceil(log2(n)) + 1` nodes, proves tree N+1 is an extension of tree N

## Roles

- **Monitor** — inspects every new entry, verifies STH
- **Auditor** — verifies append-only behavior + consistency across query sources, MMD compliance

## Gap

No normative gossip protocol — direct auditing only. Split-view detection requires out-of-band gossip, which the RFC does NOT specify. (See contrarian source 6 on why this matters for small fleets.)

## Why it matters

Every later transparency log (Rekor, KeyTrans, SCITT, sigsum, Trillian) cites RFC 9162. The leaf/node hashing rules and STH structure are load-bearing for any "append-only audit log" implementation. If you build one, copy these constants exactly.
