# The Update Framework (TUF) Specification

The most battle-tested rotation playbook for long-lived roots. The `spec_version` + monotonic `version` + dual-signed-chain pattern is directly transferable to edge device root-of-trust.

## Encoding

**Canonical JSON** (OLPC subdialect) — the rare modern spec that *does* use canonical JSON, because root metadata must be byte-stable across re-serialization. (Note: this is a known footgun per contrarian-5 — but TUF's discipline around it works.)

## Root envelope

```
{"signed": {...}, "signatures": [{"keyid": "...", "sig": "..."}]}
```

Inside `signed`:
- `_type` — role
- `spec_version` — semver string ("1.0.0") tracks the schema
- `version` — monotonic int, clients use to chain rotations
- `expires` — ISO 8601
- `keys`, `roles`

## Two version fields, distinct purposes

| Field | Type | Purpose |
|---|---|---|
| `spec_version` | semver string | schema evolution |
| `version` | monotonic int | rotation counter |

## Content-addressed keyids

`keyid = SHA-256(canonical JSON of public key)` — keyids are content-addressed, not chosen by humans. **They cannot collide or be spoofed across rotations.** Same trick SCRAPI recommends via COSE Key Thumbprint (RFC 9679).

## Root rotation chain

Version N+1 must be signed by **both** a threshold of N's keys AND a threshold of N+1's own keys. Clients walk the chain incrementally; this is the canonical "dual-signing transition" pattern at metadata granularity.

## Threshold rule

Duplicate `keyid` entries in the signatures array count once — prevents threshold inflation by replaying the same signature.

## Why it matters

Pattern transfers directly to edge fleet identity:
- The "root" = the fleet identity directory
- Each version = one rotation round
- Dual-signed chain = clients can trust new keys without out-of-band root trust update
