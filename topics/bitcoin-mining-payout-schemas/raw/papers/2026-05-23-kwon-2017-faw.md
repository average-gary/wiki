# FAW — Fork After Withholding (Kwon et al. CCS 2017)

Strict generalization of BWH. Resolves the "miner's dilemma" of pure BWH (which is a symmetric prisoners' dilemma) by giving the larger pool a structural advantage.

## Formal model

Attacker with hashrate share α joins target pool with infiltration fraction τ:
- Mines `(1-τ)α` innocently (publishes blocks normally).
- Mines `τα` inside the target pool. On finding an FPoW (full proof-of-work) via infiltration, **withholds it**.
- If an external honest miner publishes a block, the attacker **immediately submits the withheld FPoW** to the pool manager → fork.
- Pool propagates the withheld FPoW; with probability `c` (network propagation parameter) the attacker's branch wins.
- If no external block arrives during the round, the FPoW is discarded (BWH-equivalent fallback).

## Profit (Theorem 5.1)

`R_a(τ) = (1−τ)α/(1−τα) + [β/(1−τα) + cτα(1−α−β)/(1−τα)] · τα/(β+τα)`

Where β = victim pool size minus infiltrator share.

**Theorem 5.2**: FAW reward ≥ BWH reward always. Equal when c=0 (no fork-winning probability); strictly greater otherwise.

## Worked numbers

α = β = 0.2:
- c = 0 (BWH-equivalent): RER = **1.14%**
- c = 0.5: RER = **1.74%**
- c = 1: RER = **3.75%**

Multi-pool attack with F2Pool-class attacker (~20%): **56.24% more reward than BWH** (4.63% vs 2.96% RER).

## Resolution of the miner's dilemma (Theorems 7.1–7.2)

Unlike BWH (which is a symmetric prisoners' dilemma), FAW becomes a **"pool size game"**:
- Larger pool wins extra reward at unique Nash equilibrium.
- Smaller pool can be **net-loser** while larger pool stays profitable.

**This is the paper's headline novelty.** It explains why a Foundry-class pool could rationally launch FAW against AntPool-class pools and profit, even where pure BWH would zero out via mutual sabotage.

## Payout-scheme dependency

Paper assumes **uniform proportional distribution**. **Does not separately analyze PPS / PPLNS / FPPS.**

The vulnerability is a property of any scheme that pays for shares without verifying full-PoW reveal — i.e., **all standard pool schemes including FPPS, PPS+, PPLNS, TIDES, SLICE.**

## Where the operator-vs-miner incidence shifts

A subtle point not formalized in the paper but implicit: the **incidence** of FAW damage differs by scheme even though attacker profit doesn't:

- **FPPS / PPS**: pool operator absorbs the loss (pays out for shares regardless of whether real blocks land).
- **PPLNS / proportional / TIDES / SLICE**: miners absorb the loss (pool only pays when blocks confirm; sabotage = no payout).
- **eHash (hashpool)**: open question — issuance is per share, but redemption is per block.

So FAW *profit to the attacker* is scheme-invariant, but FAW *who gets hurt at the victim pool* is scheme-dependent. **Worth flagging as an open observation in the wiki**, not a cited result.

## Mitigation

- Two-phase PoW (managers solve a puzzle so miners can't recognize FPoWs). Requires hard fork.
- Decentralized pools (SmartPool, P2Pool, p2poolv2). Not widely adopted.
- Fork-rate detection. Observational only.

Authors conclude: *"finding a cheap and efficient countermeasure remains an open problem."*

**No deployed mitigation as of 2017; still open as of 2026.**
