# Plan: Phase 2-A â€” finish the in-process share-chain integration

## Context

Phase 1 (PRs #23-#28) shipped the structural skeleton: `P2poolV2Engine` with a full `JobValidationEngine` trait impl, a `Pool` runtime that mirrors `PoolSv2::start` line-for-line with our engine swapped in, a CLI binary that loads `PoolConfig` from TOML, and a `sv2-p2pool-testenv` skeleton.

**The engine doesn't yet do anything p2pool-specific.** Trait methods do structural validation only â€” `prev_hash`/`nbits` are stub-zeros, `handle_push_solution` records a synthetic share-hash, and there's no `ChainStoreHandle` / `BitcoindLike` / `ShareValidator` plumbed in. The binary boots and listens but cannot validate against an actual share chain.

Phase 2-A wires the share-chain backends into the engine and replaces the stubs with real validation, ending with a regtest E2E test.

**Phase 2-B** (capnp IPC) is deferred. Even Phase 2-B requires Phase 2-A's work on the server side.

## Decisions

1. **Scope**: Phase 2-A only.
2. **Block reconstruction helper**: in our engine crate. No upstream PRs.
3. **Execution**: iterative in-conversation. Per-phase PRs with auto-merge on green CI.
4. **Builder shape**: `EngineHandles` struct passed to a new `with_handles` constructor. Two-constructor API keeps existing tests green during Phase 2.
5. **Bitcoin tip metadata source**: bitcoind GBT (more current than p2poolv2's cached `bitcoin_header`).
6. **Process model**: `Pool::start` boots p2poolv2 `Node` in-process. Phase 2-B will revisit.
7. **p2poolv2 config**: separate `--p2pool-config` CLI flag pointing at p2poolv2's TOML. Don't extend sv2-apps's `PoolConfig`.

## Risks

- **Block reconstruction merkle math**: golden-vector test catches mismatches. Mitigation: lean on bitcoin-rs helpers; don't roll our own SHA256d.
- **`prev_hash`/`nbits` capture race**: bitcoind tip may advance between GBT and JDS forwarding. Mitigation: mirror upstream's `validation_context_drifted` pattern.
- **p2poolv2 Node port collisions** in parallel tests. Mitigation: testenv uses OS-allocated free ports.
- **AGPL boundary**: Phase 2 keeps p2poolv2_lib in-process; pool stays AGPL. Phase 2-B revisits.
- **p2poolv2 TOML opacity for sv2-apps operators**. Mitigation: heavily-commented example config.
