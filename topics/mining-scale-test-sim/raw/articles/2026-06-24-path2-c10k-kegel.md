# C10K problem (Kegel, updated through ~2014)

Foundational reference for high-concurrency server design on Linux/BSD.
Quoted heavily by every "how many connections per box" discussion.

## Kegel's per-client cost model

At 20,000 clients on a 1 GHz / 2 GB box:
- **50 KHz per client** (CPU cycles)
- **100 KB per client** (RAM)
- **50 Kbit/s per client** (bandwidth)

A 4 GHz × 16-core modern box has ~1280x the CPU budget, so the
**CPU** is essentially never the C10K wall — it's a C10M conversation
now.

## Five I/O strategies

1. select/poll (level-triggered nonblocking) — FD_SETSIZE caps
2. **epoll/kqueue (edge-triggered nonblocking)** — current standard
3. AIO with completion notification (`io_uring`, Windows IOCP)
4. Thread-per-client blocking I/O
5. In-kernel servers (Tux, etc.)

## Bottlenecks Kegel calls out

- **Thread stacks**: 2 MB × N threads = stack exhaustion before fd
  exhaustion at scale ⇒ async/coroutine-based servers win.
- **FD_SETSIZE** caps select(), not poll() or epoll().
- **Disk I/O blocks even with nonblocking sockets** — entire process
  freezes during page faults under memory pressure.
- **Poll cost** scales with idle fd count; epoll only walks ready fds.

## Modern translation for stratum pool servers

- Stratum has tiny messages (~100-200 B / share submit), very long-lived
  connections, low per-connection bandwidth — Kegel's 50 Kbit/s/client
  is an order of magnitude too high for stratum. Stratum is closer to
  **5-10 Kbit/s per active miner** including job updates.
- Per-client RAM in modern Rust/C: ~1-10 KB, vs Kegel's 100 KB estimate
  (cheap allocators + per-task green threads).
- Per-client CPU at modern Stratum V2 vardiff defaults: <50 KHz steady-
  state, MUCH more during handshake (see SRI noise-sv2 benches:
  ~317 µs of CPU per Noise_NX handshake).

## What this implies for the scale-test simulator

- The **C10K-class** problems (select/poll scaling, thread stacks) are
  solved problems for any modern Rust/Tokio or Go stratum server.
- The **C10M-class** problems (kernel memory per socket, conntrack table,
  ephemeral port exhaustion, per-syscall overhead) are the next wall —
  see Cloudflare & cilium articles in this raw corpus.
- Stratum's small-message profile makes it MORE about syscall/scheduling
  overhead per share than bandwidth.
