# SLSA levels and Nix

SLSA (Supply-chain Levels for Software Artifacts) v1.0 is the lingua franca
regulators and corporate buyers use. Important to avoid the common "we use
Nix so we're SLSA L3" overclaim.

## The levels (Build track only — Source track was removed in v1.0)

| Level | Requirement |
|-------|-------------|
| **Build L1** | Provenance exists. |
| **Build L2** | Signed provenance, generated by hosted build platform. |
| **Build L3** | Hardened build platform; secrets unreachable from user build steps; non-falsifiable provenance. |

**Hermetic ≠ L3.** SLSA v1.0 explicitly notes hermeticity is *not* an L3
requirement — only future work.

## How Nix maps

| Setup | Approx tier |
|-------|-------------|
| Nix on a developer laptop | **L1** (provenance can exist; not platform-attested) |
| Nix on Hydra / Garnix / Cachix-as-builder | **L2** (signed provenance, isolated builder) |
| Above + signing-key isolation + non-falsifiable provenance | **L3** |

Nix's hermetic sandbox *exceeds* SLSA on hermeticity but doesn't grant L3 by
itself — the *platform* still has to be hardened.

## Implication for Bitcoin

When a Bitcoin operator or auditor asks "is my Nix-built `bitcoind` SLSA
L3?", the honest answer:

> **Depends on the build platform, not on Nix-the-language.** Cachix +
> Hydra-style hosted builders with key isolation can hit L3; ad-hoc developer
> machines cannot.

Bitcoin Core's `guix.sigs` regime is technically a hybrid: each builder is
L1 individually, but the **multi-builder consensus** provides a stronger
property than any single L3 builder ([[multi-builder-attestation.md]]).

## Caveat post-xz

Source track removal in v1.0 means "SLSA-compliant" doesn't say anything
about source integrity. The xz attack ([[../../raw/articles/2026-06-15-xz-cve-2024-3094-cox.md]])
would have shipped at any SLSA level with a poisoned tarball.
