# RFC 8392 — CWT

The IETF "JWT-but-CBOR" for byte-tight contexts.

## Claims map (integer keys vs JWT string keys)

| Claim | JWT string | CWT integer |
|-------|-----------|-------------|
| iss   | "iss"     | 1 |
| sub   | "sub"     | 2 |
| aud   | "aud"     | 3 |
| exp   | "exp"     | 4 |
| nbf   | "nbf"     | 5 |
| iat   | "iat"     | 6 |
| cti   | "cti"     | 7 (CWT ID) |

CBOR encodes integers in 1-2 bytes; JWT string keys take 5+ bytes each. → **2-3× size reduction** for equivalent claims.

## Protection modes (via COSE — RFC 8152)

| Mode | Use |
|------|-----|
| `COSE_Sign1` | Asymmetric signature (Ed25519, ECDSA, etc.) |
| `COSE_Mac0`  | Symmetric MAC (HMAC, AES-CMAC) |
| `COSE_Encrypt0` | AEAD encryption (AES-GCM, ChaCha20-Poly1305) |

Modes are **nestable** — sign-then-encrypt, encrypt-then-MAC, etc.

## CoAP / IoT context

Targets constrained environments (CoAP-only stacks, sensors, embedded devices). Registered as CoAP Content-Format 61.

## Rust implementation

`coset = 0.3+` (Google, github.com/google/coset, last pushed 2026-03-02). Built on `ciborium`. Provides:

- `CoseSign1`, `CoseSign`
- `CoseMac0`, `CoseMac`
- `CoseEncrypt0`, `CoseEncrypt`
- `CoseKey`, `CoseKeySet`

CWT itself is in a dedicated module within coset.

## When to pick CWT for the iroh app token

CWT is the right primitive **only if**:

1. Your homelab AI server already speaks CBOR for other reasons (iroh-blobs / iroh-willow ecosystem leans CBOR-friendly)
2. You need byte-tight QR-printable single-use tickets where every char counts
3. You have a strong type-safety requirement that COSE structures provide

For the GTX 1060 case, **PASETO v4 wins** on developer ergonomics; Branca wins on minimal byte-budget; CWT is the right pick if you're already in CBOR-land for other reasons.
