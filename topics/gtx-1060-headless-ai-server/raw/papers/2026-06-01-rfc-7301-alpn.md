# RFC 7301 — ALPN

The TLS extension that Iroh's multi-ALPN Router rides on.

## Wire shape

- ClientHello includes ALPN extension with a list of `ProtocolName` byte strings
  - Each name is 1-255 bytes; IANA-registered; UTF-8 permitted (e.g. `http/1.1` = `0x68 74 74 70 2f 31 2e 31`)
  - Order = client preference
- Server picks one from the intersection — server preference wins on its choice from the overlap
- Server echoes in ServerHello extension
- No extra round-trips beyond the regular TLS 1.2/1.3 handshake

## Failure mode

- Empty/truncated names prohibited
- No overlap → fatal `no_application_protocol` alert (TLS handshake fails cleanly)
- Selection is binding for the connection until renegotiation

## How Iroh uses this

In TLS 1.3 (which QUIC uses, RFC 9001) the same extension carries through. `iroh::protocol::Router::accept(&ALPN, handler)` registers a handler keyed by an ALPN byte string; on accept, iroh "uses the ALPN to route to the correct protocol handler" (per docs.iroh.computer/concepts/protocols).

Multi-ALPN dispatch is therefore RFC 7301 selection happening inside the QUIC TLS 1.3 handshake, with server-side dispatch delegated to a Rust hashmap.

## ALPN naming convention in Iroh

`iroh/<protocol>/<version>` (e.g., `iroh/blobs/0`). For application-specific protocols, single-segment forms like `DUMBPIPEV0` (dumbpipe) or `sv2/0` (Stratum) are common. See [[2026-06-01-iroh-router-protocolhandler-docs]].
