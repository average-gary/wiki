# RFC 8725 — JWT BCP

IETF authority for "here are the JWT pitfalls and how to avoid them." Citable section numbers for the wiki article's "why we don't use plain JWT" section.

## The five named traps

| Trap | RFC 8725 § | Mitigation |
|------|-----------|------------|
| `alg=none` | §2.1, §3.2 | Reject unless explicitly requested |
| RS256/HS256 key confusion | §3.1 | Bind each key to exactly one algorithm |
| Weak HMAC keys | §2.2, §3.5 | No human-memorizable passwords as MAC keys |
| `kid` injection (SQL/LDAP) | §2.9, §3.10 | Validate/sanitize the kid before lookup |
| Cross-JWT confusion | §2.8, §3.12 | Mutually exclusive validation: typing, claims, keys, issuers |

## How PASETO/Biscuit avoid these

| Trap | PASETO v4 | Biscuit v3 |
|------|-----------|------------|
| `alg=none` | Header is fixed `v4.local.` / `v4.public.` — no negotiation | No alg negotiation; format version is integer |
| RS256/HS256 confusion | One algorithm per version | One algorithm (Ed25519 default, ECDSA P-256 alt) |
| Weak HMAC keys | Spec mandates 32 bytes; KDF expects randomness | Keys are pubkeys, not derived from passwords |
| `kid` injection | Footer is authenticated; kid lookup is an app concern | root_key_id is u32, not a string |
| Cross-JWT confusion | Implicit assertion binds context | Datalog policies are scoped to the verifier |

## Direct application to the iroh app token

If we follow Biscuit or PASETO instead of rolling our own:

- We **automatically dodge** the alg=none and key-confusion classes
- We still must handle kid-equivalent validation for our own scheme (PASETO footer parsing, Biscuit root_key_id selection) — sanitize the integer / known-value lookup
- We still must enforce **cross-token confusion**: ensure a token issued for service A cannot be replayed at service B. PASETO's implicit assertion (bind to NodeId or ALPN) is the seam.

## Direct application if we DO roll our own (minimal HMAC scheme)

Apply RFC 8725 mitigations explicitly:

- No algorithm field on the wire — hardcode BLAKE3 keyed-hash
- Each token type has its own key (`key_for(token_type)`)
- All token-specific keys are 32 random bytes (not user-supplied)
- Key id is an integer epoch, not a free-form string
- `(token_type, audience)` is in the MAC input → cross-token replay fails
