# BIP-327: MuSig2 for BIP340-compatible Multi-Signatures

**Authors**: Jonas Nick, Tim Ruffing, Elliott Jin. **Status**: Deployed; v1.0.3 dated 2026-01-05 (assigned 2022-03-22).

MuSig2 is explicitly **n-of-n**, "not a t-of-n threshold-signature scheme" — signing requires *all* participants that took part in key aggregation.

## Two-round structure

- **Round 1**: `NonceGen` — each signer computes a `secnonce`/`pubnonce` pair and broadcasts the `pubnonce`. Then `NonceAgg` combines all pubnonces into a single `aggnonce`.
- **Round 2**: `Sign` — each signer produces a 32-byte partial signature using their `secnonce` + secret key. Then `PartialSigAgg` combines the partial signatures into a final BIP-340-compatible Schnorr signature.
- Signers are **stateful**: they generate `secnonce` in Round 1 and must persist it until the `aggnonce` arrives for Round 2.

## Data structures

- **`SecNonce` = 97 bytes**: `bytes(32, k1) || bytes(32, k2) || pk` (two nonce scalars plus the signer's own pubkey).
- **`PubNonce` = 66 bytes**: `cbytes(R*_1) || cbytes(R*_2)` (two 33-byte compressed points).
- **`SessionContext`** = tuple `(aggnonce, u, pk_1..u, v, tweak_1..v, is_xonly_t_1..v, m)` — aggregate nonce; count + list of individual pubkeys; count + list of tweaks with per-tweak x-only mode flags; and the message.

## Algorithms

`KeyAgg` (produces a KeyAgg Context: aggregate point Q, gain accumulator `gacc`, tweak accumulator `tacc`); `KeySort` (lexicographic canonical ordering of pubkeys); `NonceGen(sk, pk, aggpk, m, extra_in)`; `NonceAgg`; `GetSessionValues` (derives `(Q, gacc, tacc, b, R, e)` from a SessionContext); `Sign` (computes `s = (k1 + b·k2 + e·a·d) mod n`); `PartialSigVerify` (checks `s·G = R*_e + e·a·g'·P`); `PartialSigAgg`.

## Nonce generation and its binding inputs

`NonceGen` takes optional `sk`, `aggpk`, `m`, `extra_in`. Binding these is defense-in-depth: *"If the value rand' was identical in two NonceGen invocations, but any other argument was different, the secnonce would still be guaranteed to be different."* The spec mandates a high-quality RNG and bans determinism: *"In contrast to BIP340 signing, the values k1 and k2 must not be derived deterministically"* — otherwise active adversaries can trick the victim into reusing a nonce.

## Critical MUST/SHOULD requirements (quoted)

- *"The Sign algorithm must not be executed twice with the same secnonce. Otherwise, it is possible to extract the secret signing key from the two partial signatures output by the two executions of Sign."*
- An implementation *"may securely erase the secnonce argument by overwriting it with 64 zero bytes after it has been read by Sign."* An all-zero secnonce is invalid and causes Sign to fail.

## Identifiable abort

`PartialSigVerify` lets an honest party attribute blame: if contributions are untampered and each partial sig is verified, and exactly one fails, *"the algorithm run by the honest party will output the index of exactly one malicious signer."* Note that individual partial sigs are forgeable in isolation; the aggregate is valid iff all partials verify.

## Tweaking

Both plain/EC tweaking (`P + t·G`, BIP32-style) and x-only tweaking (`with_even_y(P) + t·G`, BIP341/Taproot) are handled inside the session via the `tweak` / `is_xonly_t` fields of the SessionContext. Ships a Python reference implementation (explicitly "not for production").
